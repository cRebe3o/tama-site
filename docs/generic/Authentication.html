📚 Livello Base

Cos'è l'autenticazione JWT?

JWT (JSON Web Token) è un sistema per autenticare gli utenti. Funziona così:

  1. L'utente accede con email e password (JWT) o con Microsoft (MSAL)
  2. Il server verifica le credenziali o il token Azure
  3. Il server crea un JWT interno — un "biglietto" firmato che contiene l'identità dell'utente
  4. Il client riceve il JWT e lo usa per fare richieste autenticate
  5. Il server verifica il JWT su ogni richiesta — se valido, consente l'accesso

Perché JWT?

  • ✅ Stateless: il server non deve ricordare nulla sui client
  • ✅ Scalabile: funziona bene anche con molti server
  • ✅ Sicuro: il JWT è firmato digitalmente
  • ✅ Mobile-friendly: facile da usare nelle app mobili
📌 Nota: Indipendentemente dalla strategia scelta (JWT o MSAL), il backend genera sempre JWT interni. Con MSAL, il token Azure AD serve solo per il login iniziale.

🔄 Flussi Principali

1️⃣ Flusso di Login (JWT — Email+Password)

L'utente invia email e password, riceve JWT.

📱 BROWSER / APP
  • Email: mario@example.com
  • Password: ****
⬇️ POST /api/auth/login
⚙️ SERVER - LoginHandler.cs
  • ✓ Cerca utente per email
  • ✓ Verifica password (BCrypt)
  • ✓ Verifica account attivo
  • ✓ Risolve permessi (ruoli diretti + gruppi)
  • ✓ Genera JWT + Refresh Token
⬇️ 200 OK
✅ RISPOSTA
  • accessToken: "eyJ..."
  • expiresAt, userId, email, displayName, language
  • permissions: ["users.read", "roles.manage"]
  • refreshToken in cookie HttpOnly

2️⃣ Flusso di Accesso Protetto

Client usa JWT per accedere alle risorse.

📱 RICHIESTA PROTETTA
  • GET /api/users
  • Authorization: Bearer eyJ...
⬇️
🔐 SERVER - JWT Middleware
  • ✓ Parse il token
  • ✓ Verifica firma (HMAC-SHA256)
  • ✓ Verifica non sia scaduto
  • ✓ Verifica Issuer e Audience
  • ✓ ClockSkew = Zero (nessuna tolleranza)
  • ✅ JWT VALIDO → Accesso consentito
⬇️
✅ DATI RICEVUTI
  • 200 OK { data }

3️⃣ Flusso di Refresh Token

JWT è scaduto? Usa il refresh token per rinnovarlo.

📱 JWT SCADUTO
  • GET /api/users → 401 Unauthorized
⬇️ POST /api/auth/refresh (Cookie: refreshToken=...)
⚙️ SERVER - RefreshTokenHandler.cs
  • ✓ Valida refresh token (esiste, non scaduto, non revocato)
  • ✓ Verifica utente esista e sia attivo
  • 🔄 TOKEN ROTATION:
  •    • Revoca vecchio token (RT-old)
  •    • Genera nuovo JWT (AT-new)
  •    • Genera nuovo RefreshToken (RT-new)
⬇️ 200 OK + Set-Cookie
✅ NUOVO TOKEN
  • accessToken: "eyJ..." (nuovo)
  • refreshToken: "wxyz..." (nuovo, ruotato)
🔒 TOKEN ROTATION (Security Best Practice):
1️⃣ Login → Crea RT-1
2️⃣ Refresh → Revoca RT-1, crea RT-2
3️⃣ Se attacker ruba RT-1 e tenta refresh → "Token not found" (revocato) ❌

4️⃣ Flusso di Logout

📱 LOGOUT REQUEST
  • POST /api/auth/logout
  • Cookie: refreshToken=...
⬇️
⚙️ SERVER - LogoutHandler.cs
  • ✓ Cerca token in MongoDB
  • ✓ Marca come revoked (revokedAt = now)
  • ℹ️ Non lancia errore se non trovato (idempotente)
⬇️ 204 No Content
✅ DISCONNESSO
  • Pulisce localStorage (accessToken, userId, etc.)
  • Cookie eliminato dal server
  • Utente DISCONNESSO ✓

5️⃣ Flusso MSAL Login (Azure AD)

Login con Microsoft/Office 365.

📱 LOGIN CON MICROSOFT
  • Utente clicca "Login con Microsoft"
  • La libreria @azure/msal-browser redireziona a login.microsoftonline.com
⬇️ Autenticazione Azure AD
📱 RICEVE idToken DA AZURE
  • Azure AD autentica l'utente (email, password, MFA se abilitato)
  • La libreria MSAL valida la firma RSA del token
  • Restituisce idToken (JWT firmato da Microsoft)
⬇️ POST /api/auth/msal-login { azureToken: idToken }
⚙️ SERVER - MsalLoginHandler.cs
  • ✓ Legge il JWT di Azure (parsing, senza rivalidare la firma)
  • ✓ Estrae email dal claim (preferred_username o email)
  • ✓ Cerca utente LOCALE per email in MongoDB
  • ✓ Verifica account attivo
  • ✓ Risolve permessi (ruoli diretti + gruppi)
  • ✓ Genera JWT INTERNO (formato Tama, non Azure)
  • ✓ Crea Refresh Token
⬇️ 200 OK + JWT INTERNO
✅ LOGGATO CON AZURE
  • accessToken: "eyJ..." (JWT interno, non Azure)
  • Usa come qualsiasi JWT normale
  • refreshToken in cookie HttpOnly
⚠️ Nota sulla validazione:
Il backend non rivalidda la firma RSA del token Azure. La validazione crittografica avviene nel frontend tramite la libreria @azure/msal-browser. Il backend si limita a leggere il token ed estrarre l'email per trovare l'utente locale. L'utente deve esistere preventivamente nel database locale.

📱 Frontend

Implementazione Effettiva

Il frontend usa Axios con interceptor, Pinia per lo stato, e servizi dedicati.

Axios Interceptor — Gestione 401

⚠️ Comportamento su 401: Il frontend effettua il logout automatico dell'utente (non il retry con refresh). È presente una guardia anti-ricorsione per evitare loop infiniti. 
// plugins/axios.ts
api.interceptors.response.use(
  res => res,
  async error => {
    if (error.response?.status === 401 && !isLoggingOut) {
      isLoggingOut = true
      try { await useAuthStore().logout() }
      finally { isLoggingOut = false }
    }
    return Promise.reject(error)
  }
)

Router Guard — Protezione Rotte

// router/index.ts
router.beforeEach(async (to) => {
  // Non autenticato + rotta protetta → /login?returnUrl=...
  if (to.meta.requiresAuth && !authStore.isAuthenticated)
    return { name: 'login', query: { returnUrl: to.fullPath } }

  // Permesso mancante → dashboard + toast warning
  if (to.meta.permission)
    if (!authStore.permissions.includes(to.meta.permission))
      return { name: 'dashboard' }
})

LoginPage — Switch Strategia

const authStrategy = import.meta.env.VITE_AUTH_STRATEGY

// Se authStrategy !== 'msal': mostra form email/password
// Se authStrategy === 'msal': mostra bottone Microsoft
// Al mount, se ritorno da redirect MSAL: auto-login con idToken

🔐 Sicurezza

✅ Cosa è Implementato

1. HTTPS Only (Production)
Il cookie è inviato solo su HTTPS in produzione (Secure = !IsDevelopment())
2. HttpOnly Cookie
Il refresh token è in un cookie HttpOnly → JavaScript non può rubarlo (XSS protection)
3. SameSite=Lax
Il browser non invia il cookie in richieste cross-site (CSRF protection)
4. Token Rotation
Ogni uso del refresh token lo revoca e ne genera uno nuovo → una-tantum
5. BCrypt Password Hashing
Le password sono hashate con BCrypt (slow, con salt) → impossibile recuperarle
6. JWT Signature (HMAC-SHA256)
Ogni JWT è firmato con HMAC-SHA256, ClockSkew = Zero → non può essere falsificato
7. Rate Limiting
Login, msal-login e refresh: max 10 richieste ogni 15 minuti → protezione brute-force
8. Messaggi di Errore Generici
"Credenziali non valide" senza specificare se email o password sbagliata (enumeration protection)
9. Account Disabilitati
Verifica IsActive in login, refresh, e msal-login
10. Error Log Automatico
Errori 5xx inviati automaticamente al backend per monitoraggio (fire-and-forget)

🐛 Troubleshooting

❌ "401 Unauthorized — Token expired"

Causa: Il JWT è scaduto (oltre 60 minuti).

Il frontend intercetta automaticamente il 401 ed effettua il logout dell'utente. Per rinnovare preventivamente, usare authStore.refresh().

❌ "401 — Refresh token not found"

Cause:

  • Cookie non inviato (verifica SameSite, Secure, withCredentials: true)
  • Token revocato (logout o rotation)
  • Token scaduto (> 7 giorni)

❌ "Nessun utente locale con email" (MSAL)

L'utente deve essere creato in Tama prima di poter fare login con Azure AD.

❌ "Token Azure non valido" (MSAL)

Verificare che il frontend stia inviando l'idToken e non l'accessToken di MSAL.
