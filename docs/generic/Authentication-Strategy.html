Single Sign-On ❌ No ✅ Sì

Che Succede Dietro le Quinte

Entrambe le strategie fanno la stessa cosa al backend:

STRATEGIA SCELTA (JWT o MSAL) │ ▼ ┌─────────────────────┐ │ Server Backend │ ├─────────────────────┤ │ ✓ Autentica utente │ │ ✓ Risolve permessi │ │ ✓ Genera JWT interno│ │ ✓ Crea Refresh Token│ └─────────────────────┘ │ ▼ Ritorna JWT + Refresh │ ▼ Client usa JWT interno per tutte le richieste
⚠️ Nota fondamentale:
Il backend genera sempre JWT interni, indipendentemente da quale strategia usi. Con MSAL, il token Azure AD serve solo per il login iniziale per identificare l'utente. Poi il backend emette un JWT locale che il client usa normalmente.
ℹ️ Endpoint sempre attivi:
Il backend non disabilita endpoint in base alla strategia configurata. Sia /api/auth/login che /api/auth/msal-login sono sempre disponibili. È il frontend che decide quale UI mostrare.

Strategia JWT

📧 Email + Password (Classica)

Configurazione Backend

{
  "Auth": {
    "Strategy": "Jwt",
    "Jwt": {
      "Secret": "your-super-secret-key-at-least-32-characters",
      "Issuer": "Tama",
      "Audience": "TamaClients",
      "ExpiresInMinutes": 60,
      "RefreshExpiresInDays": 7
    }
  }
}

Configurazione Frontend

VITE_API_BASE_URL=http://localhost:5001
VITE_AUTH_STRATEGY=jwt

File Backend Principali

  • Features/Auth/Login/LoginHandler.cs — logica di login
  • Features/Auth/Shared/JwtTokenHelper.cs — generazione JWT
  • Features/Auth/Login/LoginValidator.cs — validazione input (FluentValidation)

Come Funziona Tecnicamente

  • Password: verificata con BCrypt (one-way hash, slow)
  • JWT: firmato con HMAC-SHA256
  • Claims: sub, email, name, jti, permissions[]
  • Scadenza: 60 minuti (configurabile)
  • Refresh: ogni ora ottieni un nuovo JWT
✅ Vantaggi:
• Setup semplice (5 minuti)
• Nessuna dipendenza da Azure
• Completo controllo su password policy
• Funziona offline/on-premise
• Debug facile (tutto nel tuo codice)
❌ Svantaggi:
• Gestione password manuale
• Nessun SSO
• 2FA/MFA: implementare da zero
• Password reset: implementare da zero

Strategia MSAL

☁️ Azure AD (Microsoft/Office 365)

Configurazione Backend

{
  "Auth": {
    "Strategy": "Msal",
    "Jwt": { "Secret": "..." },
    "Msal": {
      "Instance": "https://login.microsoftonline.com/",
      "TenantId": "your-azure-tenant-id",
      "ClientId": "your-azure-app-client-id"
    }
  }
}

Configurazione Frontend

VITE_API_BASE_URL=http://localhost:5001
VITE_AUTH_STRATEGY=msal
VITE_MSAL_CLIENT_ID=your-azure-app-client-id
VITE_MSAL_TENANT_ID=your-azure-tenant-id

Dove Ottenere TenantId e ClientId

  1. Vai su Azure Portal → Azure Active Directory
  2. TenantId: Home → Gestisci → Proprietà → ID directory
  3. ClientId: Registrazioni app → Seleziona app → ID applicazione (client)

File Backend Principali

  • Features/Auth/MsalLogin/MsalLoginHandler.cs — parsing token Azure + lookup utente

Come Funziona Tecnicamente

  • Token Azure: firmato con chiave RSA di Microsoft, validato da @azure/msal-browser nel frontend
  • Backend: legge il token ricevuto (parsing JWT, senza rivalidare la firma)
  • Email estrapolata: dal claim preferred_username o email
  • Utente locale: cercato in MongoDB per email corrispondente
  • JWT interno: generato come JWT standard (stesso della strategia JWT)
⚠️ Nota sulla validazione:
Il backend non rivalidda la firma RSA del token Azure. La validazione crittografica avviene nel frontend tramite la libreria @azure/msal-browser. Il backend si limita a estrarre l'email per trovare l'utente locale.
✅ Vantaggi:
• SSO con Office 365, Teams, Outlook
• 2FA/MFA gestito da Microsoft
• Niente gestione password
• Conditional Access (Azure AD P1/P2)
• Compliance built-in (SOC 2, ISO 27001)
❌ Svantaggi:
• Richiede Azure AD Tenant
• Setup più complesso (Azure Portal)
• Dipendenza da Microsoft services
• Doppia gestione utenti (Azure AD + DB locale)
• Debug più difficile

Flussi Completi

Flusso 1: Login JWT

CLIENT: POST /api/auth/login { email, password } │ ▼ SERVER AUTENTICA • Verifica email • Verifica password BCrypt • Risolve permessi (diretti + gruppi) • Genera JWT interno (HMAC-SHA256) • Salva refresh token (MongoDB) │ ▼ 200 OK { accessToken: "eyJ...", expiresAt: "...", userId, email, displayName, language, permissions: [...] } Set-Cookie: refreshToken=... (HttpOnly) │ ▼ BROWSER RICEVE JWT • Salva in localStorage • Riceve refreshToken in cookie • LOGGATO ✓

Flusso 2: Login MSAL

BROWSER: msalInstance.loginRedirect() │ ▼ AZURE AD: login.microsoftonline.com • Utente inserisce credenziali Microsoft • MFA se abilitato • Genera idToken (JWT firmato RSA) │ ▼ BROWSER: Redirect a /login con idToken • @azure/msal-browser valida la firma │ ▼ CLIENT: POST /api/auth/msal-login { azureToken: idToken } │ ▼ SERVER: • Legge JWT Azure (parsing, no validazione firma) • Estrae email (preferred_username / email) • Cerca utente locale per email • Risolve permessi (diretti + gruppi) • Genera JWT INTERNO (formato Tama) • Salva refresh token │ ▼ 200 OK (stessa response di JWT) Set-Cookie: refreshToken=... │ ▼ Da qui in poi: IDENTICO a JWT

Flusso 3: Refresh + Rotation

CLIENT: POST /api/auth/refresh Cookie: refreshToken=RT-1 │ ▼ SERVER: • Trova RT-1 nel DB → esiste, non scaduto • Verifica utente attivo • REVOCA RT-1 (token rotation) • Risolve permessi aggiornati • Genera nuovo JWT (AT-2) • Crea nuovo refresh token (RT-2) │ ▼ 200 OK { accessToken: AT-2, expiresAt, permissions } Set-Cookie: refreshToken=RT-2

Flusso 4: Logout

CLIENT: POST /api/auth/logout Cookie: refreshToken=... │ ▼ BACKEND: Revoca refreshToken Cancella cookie │ ▼ 204 No Content │ ▼ CLIENT: Pulisce localStorage Redirect a / Utente DISCONNESSO ✓

Come Switchare

Da JWT → MSAL

Passo 1: Configura Azure AD

  1. Azure Portal → Registra una nuova app
  2. Annota: TenantId e ClientId
  3. Configura redirect URI: http://localhost:5173/login

Passo 2: Aggiorna Backend

{
  "Auth": {
    "Strategy": "Msal",
    "Msal": {
      "TenantId": "your-tenant-id",
      "ClientId": "your-client-id"
    }
  }
}

Passo 3: Aggiorna Frontend

VITE_AUTH_STRATEGY=msal
VITE_MSAL_CLIENT_ID=your-client-id
VITE_MSAL_TENANT_ID=your-tenant-id
📌 Risultato:
• /api/auth/msal-login — usato dal frontend
• /api/auth/login — sempre disponibile ma non usato dal frontend MSAL
• LoginPage mostra solo bottone "Login con Microsoft"

Da MSAL → JWT

Cambia Strategy a "Jwt" nel backend e VITE_AUTH_STRATEGY=jwt nel frontend. Le variabili MSAL possono essere rimosse o lasciate.

⚠️ Nota importante:
Il backend non disabilita nessun endpoint. Entrambi i percorsi (/login e /msal-login) sono sempre disponibili. La configurazione Auth:Strategy serve per documentazione. È il frontend che sceglie quale UI mostrare in base a VITE_AUTH_STRATEGY.

Confronto Dettagliato

Caratteristica JWT MSAL
Credenziali Email + Password locale Account Microsoft/Azure AD
Storage credenziali MongoDB (hash BCrypt) Azure AD (cloud Microsoft)
Login UI Form email/password Bottone Microsoft → redirect
SSO ❌ No ✅ Office 365, Teams, etc.
MFA ❌ Da implementare ✅ Microsoft Authenticator, SMS, FIDO2
Password reset ❌ Da implementare ✅ Gestito da Microsoft
Pre-requisiti Utente nel sistema Utente in Azure AD + nel sistema
Dipendenze esterne Nessuna Azure AD, Microsoft Identity
Token per API JWT interno (identico) JWT interno (identico)
Refresh token Rotation + HttpOnly cookie Rotation + HttpOnly cookie
Rate limiting 10 req / 15 min 10 req / 15 min

🔑 Quando scegliere JWT

  • App interna o prototipo
  • Team piccolo senza Office 365
  • Massimo controllo sul flusso auth
  • Nessun tenant Azure AD
  • Deploy on-premise/offline

☁️ Quando scegliere MSAL

  • Azienda con Office 365 / Microsoft 365
  • Servono SSO e MFA senza implementarli
  • Azure AD tenant già configurato
  • Ecosistema enterprise Microsoft
  • Requisiti di Conditional Access

Sicurezza

✅ Implementato (comune a entrambe)

HTTPS Only (Prod) · HttpOnly Cookie · SameSite=Lax · Token Rotation · HMAC-SHA256 · Rate Limiting · Generic Error Messages · Account Disable Check

JWT-specific

✅ BCrypt password hashing (slow, salted)
✅ FluentValidation su LoginCommand (email + password)

MSAL-specific

✅ Token Azure validato dalla libreria @azure/msal-browser (firma RSA)
✅ Cache MSAL in localStorage (gestita dalla libreria)
✅ Redirect flow (non popup) — più sicuro
⚠️ Considerazione MSAL:
Il backend non rivalidda la firma del token Azure. La sicurezza dipende dalla validazione lato client tramite @azure/msal-browser. In produzione, valutare l'implementazione della validazione server-side con JWKS per difesa in profondità.

⚠️ Best Practices

  • Secret JWT: generare casuale (min 32 char), usare Azure Key Vault in prod
  • HTTPS: sempre in produzione (cookie Secure flag)
  • User sync MSAL: utenti devono esistere nel DB locale prima del login Azure
  • Logging: mai loggare password, token, o dati sensibili

Troubleshooting

❌ "401 Unauthorized — Token expired"

Il frontend intercetta il 401 ed effettua logout automatico. Per rinnovare preventivamente usare authStore.refresh().

❌ "Nessun utente locale con email" (MSAL)

L'utente deve essere creato in Tama prima di fare login con Azure AD.

❌ "Token Azure non valido" (MSAL)

Verificare che il frontend invii l'idToken, non l'accessToken di MSAL.

❌ Bottone Microsoft non appare

Causa: VITE_AUTH_STRATEGY è diverso da msal (il confronto è case-sensitive).

❌ Form login non visibile

Causa: VITE_AUTH_STRATEGY=msal. Cambiare a jwt per mostrare il form.
